Lazy Reading via readHeadAndTail()

Reads first/last N bytes via 64KB-256KB buffers. Extracts JSON fields without loading the full file. Used for loading most recent messages before a full transcript read. Sessions can grow to multiple GB.

Network Layer
Transport Protocols & Resilience
Three transport implementations with reconnection, echo dedup, sleep detection, and serialized batch uploading.
🔌
WebSocket Transport
Full duplex with ping/pong, sleep detection, last-sent-id dedup
Ping: 10sKeep-alive: 5minGive-up: 10min
  • Dual runtime: Bun (native WebSocket) and Node.js (ws package) with runtime detection
  • Liveness: Ping every 10s, if no pong on next ping → force reconnect
  • Reconnect: Exponential backoff 1s → 30s, ±25% jitter, 10min budget
  • Sleep detection: If gap > 60s between attempts → reset error budget
  • Dedup: Last-sent-id tracking, X-Last-Request-Id header on reconnect
  • Permanent close codes: 1002, 4001 never retry. 4003 retries with refreshed headers
  • Memory leak prevention: All listeners cleaned up before each reconnect
📡
SSE Transport
Server-Sent Events with sequence tracking and liveness timeout
Liveness: 45sWrite retries: 10
  • Sequence tracking: Event ID = sequence number, high-water mark for resumption
  • Dedup set: Tracks seen sequences with pruning (keep 200 around high-water)
  • Last-Event-ID header: Sent on reconnect for server-side resumption
  • Liveness: Any frame (including SSE comments) resets 45s timer
  • Write path: HTTP POST with 10 retries, exponential backoff 500ms → 8s
  • Permanent 4xx (non-429): No retry, drop message
🔀
Hybrid Transport
WebSocket reads + HTTP POST writes with SerialBatchEventUploader
Batch: 100ms flushMax queue: 100KGrace: 3s close

Write Flow

write(stream_event) → buffer (100ms BATCH_FLUSH_INTERVAL) → enqueue → SerialBatchEventUploader → POST /session/{id}/events write(other) → flush buffered stream_events first // ordering guarantee → enqueue → postOnce() [serial retry loop]

SerialBatchEventUploader

  • At most one POST in-flight at any time (serialized)
  • Quadratic backoff: 500ms base, 8s max, 1s jitter
  • Max batch size: 500 events per POST
  • Max queue: 100,000 events (memory bound)
  • On-batch-dropped callback for telemetry
  • Graceful shutdown: 3s grace period for queued writes on close()

<system-reminder> Injection Mechanism
How context is injected into tool results and user messages without disrupting the conversation

<system-reminder> XML tags are the injection mechanism for runtime context. They appear inside tool results and user messages but are actually system-level information.

What Gets Injected via system-reminder

  • Git status + CLAUDE.md — First user message (isMeta=true)
  • Available skills — "Skills relevant to your task: ..." in tool results
  • Agent definitions — Available agent types for AgentTool
  • Plan mode reminders — Planning constraints and instructions
  • LSP diagnostics — Language server warnings/errors
  • Async hook responses — Background hook results
  • Memory attachments — Nested memory file contents
  • Deferred tool schemas — Lazy-loaded tool definitions from ToolSearch

The system prompt tells the model: "Tool results and user messages may include <system-reminder> tags. They contain useful information from the system and bear no direct relation to the specific tool results in which they appear."

Multi-Agent System
Agent Spawning, IPC & Isolation
Agents run in the same Node.js process (not separate processes). Communication is via task notification messages and disk output files.

Process Model — No Separate Processes
Agents run in the parent's event loop via async closures. Worktrees provide filesystem isolation

Three Execution Modes

ModeProcess ModelBlocks Parent?Trigger
SyncIn-process, main event loopYes (until auto-background at 2s)Default when run_in_background: false
AsyncIn-process, microtask queueNo — fire-and-forget closurerun_in_background: true, coordinator mode, Kairos

iKey insight: Agents are NOT separate processes. They run as async generators in the same Node.js event loop. This means they share memory, CPU, and can't truly parallelize compute — but it enables zero-overhead context sharing and prompt cache reuse.

Context Inheritance

// What child inherits from parent: ✅ Tool pool (options.tools) ✅ MCP clients (options.mcpClients) ✅ File state cache (deep clone for isolation) ✅ Permission mode (or overridden by agent definition) ✅ System prompt bytes (identical copy for cache hits) // What's isolated: ❌ readFileState → deep clone ❌ abortController → new child (parent abort propagates DOWN, not UP) ❌ setAppState → no-op (breaks mutation isolation) ❌ messages → cloned or overridden ❌ nestedMemoryAttachmentTriggers → fresh empty Set

Fork Subagent — Cache-Identical Spawning

When subagent_type is omitted, the child inherits the parent's exact system prompt bytes and full conversation context. This means the API request prefix is byte-identical to the parent's, enabling prompt cache hits. A <fork-boilerplate-tag> is appended to prevent recursive forking.

Communication (No IPC — Message Queue)

  • Parent → Child: Initial prompt + context at spawn time. No mid-execution messages
  • Child → Parent: Task notification XML enqueued to parent's message queue on completion
  • Agent → Agent (teams): Mailbox files at ~/.claude/teams/{team}/mail/{recipient}.jsonl
  • Output storage: JSONL transcript at ~/.claude/agents/{sessionId}/{agentId}.jsonl

Auto-Mode Classifier (YOLO) — ML-Based Permission Decisions
Calls Claude to classify whether a tool use is safe to auto-approve

When permission mode is auto, every tool call that would normally prompt the user is instead sent to an ML classifier that decides allow/deny/ask.

Classification Flow

1
Load classifier prompt

Pre-built prompts from /yolo-classifier-prompts/ — system prompt + permission template (external vs internal)

2
Merge user rules

User's settings.autoMode allow/deny rules injected into classifier prompt

3
API call via sideQuery()

Sends tool name + input to classifier model, parses structured response

4
Decision extraction

Parses tool_use block from response: shouldBlock, reason, confidence (high/medium/low)

Two-Stage Classification

  • Fast stage: Quick pattern matching, low-confidence fallback
  • Thinking stage: Full reasoning if fast stage uncertain

Integration with Permission System

The classifier runs in parallel with the permission UI dialog (3s timeout). If it finishes with high confidence before the user clicks, the dialog is auto-resolved. This is the peekSpeculativeClassifierCheck() mechanism covered in the deep dive.

!Trust gap: The classifier is an external ML model making security decisions. There's no human review loop — a confident-but-wrong classification auto-approves the action. This is the main security concern with auto mode.
