Request Lifecycle
Anatomy of a Query
When you type a message, it travels through 8 distinct phases. Here's exactly what happens at each step.
1
User Input
Your message enters via one of three channels: terminal REPL, IDE bridge (WebSocket), or SDK (headless). The REPL uses React/Ink to capture keystrokes with vim mode support.
// Entry channels
Terminal → REPL.tsx reads stdin via Ink
IDE → bridgeMain.ts receives WebSocket msg
SDK → Headless HTTP/SSE request body
2
Input Processing
The input is tokenized — slash commands (/commit) are routed to command handlers, while natural language messages proceed to the query pipeline.
/command → commands.ts registry lookup
message → QueryEngine.ts pipeline
attachments → file content injection
paste detect → large input handling
3
Context Assembly
The system builds a rich context: git status, working directory, CLAUDE.md memory files, system prompt with tool descriptions, and conversation history.
context.ts → getUserContext()
// Collects:
• Git branch, status, recent commits
• CLAUDE.md files (project memory)
• Active MCP server tools
• Permission rules & settings
• 45+ tool JSON schemas
4
LLM API Call
The assembled payload is streamed to the Anthropic API. The engine handles prompt caching, thinking mode, token budgets, and retry logic with exponential backoff.
claude.ts → messages.create(stream)
// Features:
• Prompt caching (reuse across turns)
• Extended thinking mode + budget
• Fallback models on rate limit
• Token counting & cost tracking
5
Tool Execution
When the model emits a tool_use block, the tool loop validates the schema, checks permissions (may prompt the user), executes the tool, and captures output.
// Tool loop (may repeat many times):
1. Parse tool_use from stream
2. Validate input against Zod schema
3. useCanUseTool() → allow/deny/ask
4. Execute tool handler
5. Capture & truncate output
6. Return tool_result to API
6
Agentic Loop
The model may need multiple rounds of tool calls. The agentic loop continues streaming API responses, executing tools, and feeding results back until the model produces a final text response.
API Call → Tool Use → Tool Result → API Call → Final Text
7
Response Rendering
React/Ink components render the response in real-time: markdown formatting, syntax-highlighted code blocks, tool result displays, and progress spinners.
MessageResponse.tsx → renders markdown
ToolResult renderers → format outputs
Spinner.tsx → progress indicator
StatusLine.tsx → token/cost bar
8
State Update
The final response is stored in conversation history, costs are tracked, session state is persisted, and analytics events are logged. The REPL awaits the next input.
AppState ← messages, costs, tokens
Session ← persisted to disk
Analytics ← usage events logged
REPL → ready for next prompt

Query Engine
The brain — orchestrates conversations with Claude

The Query Engine (QueryEngine.ts + query.ts) is the core orchestrator — it implements the think → act → observe agentic loop that drives every interaction.

Key Responsibilities

  • Manages single-turn and multi-turn conversation flows
  • Builds system initialization messages with context
  • Handles token budget management and auto-compaction
  • Implements the streaming tool loop (call → execute → return)
  • Manages "thinking" blocks for extended reasoning
  • Implements retry logic for failed API calls
  • Handles max_output_tokens recovery

Agentic Loop Implementation

// Simplified agentic loop (query.ts) while (true) { // 1. Call Claude API with streaming const response = await streamAPI(messages, tools) // 2. Check if model wants to use a tool if (response.has_tool_use) { const result = await executeTool(tool_use) messages.push(tool_result) continue // Loop back for next API call } // 3. Model produced final text → done return response }

Auto-Compaction

When conversation history approaches the context window limit, the engine automatically triggers compaction — summarizing earlier messages while preserving critical context like tool results and decisions.

Tool System
45+ tools — from bash to sub-agents

Every capability Claude Code has is exposed through the tool system. Each tool is self-contained with its own schema, permissions, UI renderer, and execution logic.

Tool Structure

// Each tool defines: { name: "Bash", description: "Execute shell commands", inputJSONSchema: { ... }, // Zod v4 schema handler(args) { ... }, // Execution logic canUseTool(ctx) { ... }, // Permission check }

Tool Categories

  • File Operations — Read, Write, Edit (surgical edits), Glob (pattern match)
  • Execution — Bash (160K lines of security!), PowerShell, REPL
  • Search — Grep (ripgrep), WebSearch, WebFetch
  • Agents — AgentTool (spawn sub-agents), SkillTool, SendMessage
  • Tasks — Create, Get, List, Update, Stop background tasks
  • MCP — Invoke external server tools, list/read resources
  • Planning — EnterPlanMode, ExitPlanMode, EnterWorktree

BashTool Security Stack

The BashTool alone has 160,000+ lines of code, with dedicated modules for command security validation, permission rules, read-only enforcement, and path security checks.

Model Context Protocol
Connect to external tool servers dynamically

MCP allows Claude Code to connect to external servers that provide additional tools, resources, and prompts. It's the extension system that makes Claude Code infinitely customizable.

Transport Types

  • stdio — Direct process I/O (local servers)
  • SSE — Server-Sent Events (remote HTTP)
  • HTTP — Synchronous request/response
  • WebSocket — Full-duplex real-time
  • SDK — In-process via SDK control transport

Connection Flow

Config Load → OAuth Auth → Transport Init → Tool Discovery → Ready

Config Sources (Priority Order)

  • Local settings (.claude/settings.json)
  • Project settings (project root)
  • User settings (~/.claude/)
  • Enterprise/policy settings (managed)
  • Dynamic/managed settings (remote sync)
🛡️
Permission System
Every tool call is gated by rules

The permission system ensures Claude Code never executes an action without proper authorization. Every single tool invocation passes through useCanUseTool() (40K+ lines).

Permission Modes

  • default — Ask user for each non-safe tool call
  • bypassPermissions — Trust everything (dangerous)
  • auto — Apply rules, escalate unknowns
  • dontAsk — Deny anything that would need asking
  • plan — Read-only exploration mode

Decision Flow

?
Check Permission Rules

Match tool name + args against allow/deny patterns

✓ Allow — auto-execute
✗ Deny — block silently
? Ask — prompt user

Rule Examples

allow: Bash(git *) // Auto-approve git commands allow: FileEdit(/src/*) // Auto-approve src edits deny: Bash(rm -rf *) // Block destructive commands ask: Bash(npm *) // Prompt for npm commands

IDE Bridge
VS Code & JetBrains integration layer

The Bridge layer (bridgeMain.ts — 115K lines) enables bidirectional communication between Claude Code sessions and IDE extensions, providing a seamless coding experience.

Architecture

IDE Extension ↔ WebSocket/SSE ↔ Bridge Layer ↔ Claude Session

Key Components

  • bridgeMain.ts — Main event loop and session management
  • bridgeApi.ts — API surface for IDE extensions
  • bridgeMessaging.ts — Message serialization protocol
  • jwtUtils.ts — JWT authentication between CLI and IDE
  • sessionRunner.ts — Session lifecycle management
  • bridgeUI.ts — Bridge-specific UI rendering

Capabilities

The bridge enables: inline diff previews, file change notifications, command execution, real-time streaming, and session sharing between terminal and IDE.

💾
State Management
React context store for all app state

State is managed through a custom React Context store (AppState.tsx) using a simple getter/setter pattern with listener subscriptions — no Redux, no Zustand in the CLI.

State Shape

AppState { messages[] // Conversation history mcp // Connected servers, tools permissions // Rules, mode, context settings // User preferences plugins // Installed plugins skills // Available skills tasks // Background tasks stats // Token usage, costs ui // Theme, sidebar, etc. }

Access Pattern

  • useAppState(selector) — Subscribe to specific state slices
  • setAppState(updater) — Mutate state with updater function
  • onChangeAppState — Side-effect handlers for state changes

The Web UI (separate app) uses Zustand with localStorage persistence for client-side state, while the CLI uses this custom implementation optimized for React/Ink's rendering model.
